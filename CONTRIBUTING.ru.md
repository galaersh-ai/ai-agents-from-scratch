# Руководство по контрибьюции

Спасибо, что рассматриваете возможность вклада в AI Agents from Scratch!

## Философия проекта

Этот репозиторий учит основам AI-агентов, создавая всё с нуля. Каждый вклад должен поддерживать эту учебную миссию.

**Основные принципы:**
- **Ясность вместо хитрости** — код должен быть простым для понимания
- **Фундаментальное primero** — никаких чёрных ящиков или магии
- **Прогрессивное обучение** — каждый пример основан на предыдущем
- **Локальный приоритет** — без зависимостей от API

## Типы вкладов

### Отчёты об ошибках
Нашли что-то сломанное? Откройте issue с указанием:
- Какой пример (`intro/`, `react-agent/` и т.д.)
- Что Вы ожидали vs что произошло
- Ваше окружение (версия Node, ОС, используемая модель)
- Шаги для воспроизведения

### Улучшения документации
- Исправление опечаток и грамматики
- Более ясные объяснения
- Лучшие комментарии к коду
- Дополнительные примеры в документации
- Диаграммы и визуализации

### Новые примеры
Хотите добавить новый паттерн агента? Отлично! Пожалуйста:
1. **Сначала откройте issue** — обсудим, вписывается ли он
2. Следуйте существующей структуре:
   - `pattern-name/pattern-name.js` — рабочий код
   - `pattern-name/CODE.md` — подробное объяснение кода
   - `pattern-name/CONCEPT.md` — почему это важно, варианты использования
3. Держите просто и с хорошими комментариями
4. Тщательно тестируйте минимум с одной моделью

### Улучшения кода
- Оптимизации производительности (с бенчмарками)
- Лучшая обработка ошибок
- Более ясные имена переменных
- Более полезный вывод в консоль

## Что мы НЕ ищем

- Интеграции с фреймворками (LangChain и т.д.) — этот репозиторий учит, что они делают
- Примеры с Cloud API — держим локально
- Продакшен-фичи (мониторинг, масштабирование) — это учебный материал
- Сложные абстракции — дружелюбно для новичков

## Процесс контрибьюции

1. **Fork** репозиторий
2. **Создайте ветку**: `git checkout -b fix/issue-description`
3. **Внесите изменения** и тщательно протестируйте
4. **Закоммитьте** с ясными сообщениями: `git commit -m "Fix: clarify ReAct loop explanation"`
5. **Запушьте**: `git push origin fix/issue-description`
6. **Откройте Pull Request** с:
   - Ясным заголовком
   - Описанием того, что изменилось и почему
   - Какой issue это решает

## Стандарты кода

- Используйте ясные, описательные имена переменных
- Добавляйте комментарии, объясняющие *почему*, а не просто *что*
- Следуйте существующему стилю кода (без линтера, просто копируйте паттерны)
- Держите примеры самодостаточными (один файл, когда возможно)
- Тестируйте с моделями Qwen или Llama перед отправкой

## Стандарты документации

- Используйте ясный, простой язык
- Объясняйте концепции перед кодом
- Включайте диаграммы где уместно (ASCII-арт подходит!)
- Предоставляйте практические варианты использования
- Ссылайтесь на связанные примеры
