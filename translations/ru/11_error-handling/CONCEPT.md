## Концепция: Комплексная обработка ошибок для агентов

Агенты падают большими способами, чем обычные приложения, потому что они оркестрируют **несколько ненадёжных шагов**:

- **Вызовы LLM** (таймауты, ограничения ресурсов, некорректные выводы, ошибки выполнения)
- **Выполнение инструментов** (сетевые сбои, неверный ввод, недоступные сервисы)
- **Логика рабочего процесса** (политики-ограничения, частичное завершение, цепочки зависимостей между инструментами)

Этот пример использует три идеи для того, чтобы сбои были безопасными и понятными:

### 1) Стандартизированная таксономия ошибок

Используйте небольшой набор классов ошибок со **стабильными кодами** и согласованными полями:

- **`ValidationError`**: ввод пользователя отсутствует/невалиден (быстрое падение; обычно не повторяется)
- **`LLMCallError`**: провайдер/модель LLM упала или вернула непригодный вывод (часто повторяется)
- **`ToolExecutionError`**: инструмент упал (иногда повторяется, иногда нет)
- **`AgentWorkflowError`**: сбой **на уровне оркестрации** — многошаговый запуск не может быть завершён по проекту.

В продакшене Вы можете разделить **`AgentWorkflowError`** на более мелкие типы (политика, цепочка рабочего процесса, полный сбой системы). Этот урок сохраняет **один класс** с полем **`step`** и коротким комментарием источника, чтобы **одна и та же структура** могла представлять эти идеи в маленькой демонстрации.

Примеры в этом репозитории:

- **`policy_guard`** — после валидации ограничитель блокирует запрос (демо: пользователь упоминает **`u_demo_workflow`**).
- **`resolve_user_profile`** — в деградированном режиме **попытки восстановления основного пути исчерпаны**, **пробуется запасной вариант**, и **запасной вариант тоже падает**; ошибка выдаётся на уровне рабочего процесса с внутренней ошибкой инструмента как **`cause`**.

Каждая ошибка включает:

- **`code`**: машиночитаемый, стабильный идентификатор (хорош для метрик и алертов)
- **`userMessage`**: безопасное, нетехническое сообщение для пользователей
- **`retryable`**: подходит ли автоматический повтор
- **`details`**: структурированные данные для логов (имя инструмента, имя шага, ID и т.д.)
- **`cause`**: исходная ошибка (для сохранения цепочек корневой причины)

**`AgentWorkflowError`** дополнительно несёт **`step`**. Его `details` объединяют `step` с любой дополнительной метаинформацией, которую Вы передаёте.

### 2) Классификация и стратегии восстановления

**Нормализуйте, затем классифицируйте.** **`normalizeUnknownError`** превращает произвольные брошенные значения в **`AppError`**. **`classifyError`** добавляет **`retryable`** и **`type`** (`error.code`), чтобы повторы, логи и сообщения для пользователей использовали один конвейер вместо повторения деревьев `instanceof`.

Стратегии восстановления (типичная лестница), как показано с реальной локальной LLM через `node-llama-cpp`:

- **Таймаут**: ограничьте, как долго любой шаг может зависнуть
- **Повтор**: только когда **`classifyError`** говорит **`retryable`** (с backoff и jitter)
- **Запасной вариант**: если основной инструмент падает **временно**, запустите более безопасную альтернативу, возвращающую деградированный, но полезный ответ
- **Деградированный режим**: если путь LLM падает, делегируйте **`runDegradedProfileResolution`** — детерминированное извлечение ID **`u_<цифры>`** и та же модель **основной → запасной** инструмент, без встраивания этой логики в огромный `catch`
- **Изящное падение**: если восстановление невозможно, верните **`formatUserFacingError`** плюс correlation ID

Когда **оба** — основной (после повторов) и запасной — падают в деградированном режиме, пример повышает этот результат до **`AgentWorkflowError`**: пользователь по-прежнему видит одно ясное сообщение, а **`cause`** сохраняет базовую ошибку инструмента для отладки.

### 3) Отделение сообщений для пользователей от отладочной информации

Пользователи должны видеть:

- ясные следующие шаги (попробуйте снова, переформулируйте, сократите ввод)
- без стек-трейсов или деталей провайдера
- **reference ID**, который можно передать в поддержку

Разработчики/операторы должны видеть:

- стабильный `code` ошибки
- структурированные `details`
- correlation ID
- исходный **`cause`**

Для **`AgentWorkflowError`** пример также печатает **консольный баннер** (шаг, код, correlation ID, сообщения, details, сводка cause), чтобы живые демонстрации и локальная отладка оставались читаемыми рядом с компактной строкой лога `[agent_error]`.

### Детерминированные демо и `SIMULATION`

Чтобы учебные запуски были предсказуемыми, ID пользователей **`u_999`** и **`u_777`** управляются небольшой картой **`SIMULATION`** (см. `error-handling.js`):

- **`u_demo_workflow`** в тексте вызывает **`policy_guard`**.
- **`SKIP_LLM_DEGRADED u_777`** пропускает LLM, входит в деградированный режим и использует **`u_777`**, чтобы основной и запасной оба падали воспроизводимо.

### Почему этот паттерн масштабируется

- **Согласованность**: каждый сбой имеет одну и ту же форму; неизвестные ошибки нормализуются до обработки
- **Наблюдаемость**: метрики/алерты группируются по `code` и по шагу рабочего процесса `step`
- **Безопасность**: чувствительные/специфичные для провайдера детали не попадают в сообщения для пользователей
- **Устойчивость**: временные проблемы восстанавливаются автоматически; жёсткие сбои деградируют или выдаются как одна ошибка уровня рабочего процесса с сохранённым **`cause`**
