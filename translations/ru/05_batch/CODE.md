# Объяснение кода: batch.js

Этот файл демонстрирует **параллельное выполнение** нескольких промптов LLM с использованием отдельных контекстных последовательностей, обеспечивая параллельную обработку для повышения производительности.

## Пошаговый разбор кода

### 1. Импорт и настройка (строки 1-10)
```javascript
import {getLlama, LlamaChatSession} from "node-llama-cpp";
import path from "path";
import {fileURLToPath} from "url";

/**
 * Asynchronous execution improves performance in GAIA benchmarks,
 * multi-agent applications, and other high-throughput scenarios.
 */

const __dirname = path.dirname(fileURLToPath(import.meta.url));
```
- Стандартные импорты для взаимодействия с LLM
- Комментарий объясняет преимущества производительности
- **Бенчмарк GAIA**: Стандарт для тестирования производительности AI-агентов
- Полезно для мультиагентных систем, которым нужно обрабатывать множество запросов

### 2. Конфигурация пути к модели (строки 11-16)
```javascript
const modelPath = path.join(
    __dirname,
    "../",
    "models",
    "DeepSeek-R1-0528-Qwen3-8B-Q6_K.gguf"
)
```
- Использует **DeepSeek-R1**: модель с 8B параметрами, оптимизированная для рассуждений
- **Квантизация Q6_K**: Баланс между качеством и размером
- Модель загружается один раз и разделяется между последовательностями

### 3. Инициализация Llama и загрузка модели (строки 18-19)
```javascript
const llama = await getLlama();
const model = await llama.loadModel({modelPath});
```
- Стандартная инициализация
- Модель загружается в память один раз
- Будет использоваться несколькими последовательностями одновременно

### 4. Создание контекста с несколькими последовательностями (строки 20-23)
```javascript
const context = await model.createContext({
    sequences: 2,
    batchSize: 1024 // The number of tokens that can be processed at once by the GPU.
});
```

**Ключевые параметры:**

- **sequences: 2**: Создаёт 2 независимые последовательности разговоров
  - У каждой последовательности своя история разговоров
  - Обе разделяют одну модель и пул памяти контекста
  - Могут обрабатываться параллельно

- **batchSize: 1024**: Максимальное количество токенов, обрабатываемых за GPU-батч
  - Больше = лучше утилизация GPU
  - Меньше = меньше потребление памяти
  - 1024 — хороший баланс для большинства GPU

### Почему несколько последовательностей?

```
Single Sequence (Sequential)     Multiple Sequences (Parallel)
─────────────────────────       ──────────────────────────────
Process Prompt 1 → Response 1    Process Prompt 1 ──┐
Wait...                                              ├→ Both responses
Process Prompt 2 → Response 2    Process Prompt 2 ──┘   in parallel!

Total Time: T1 + T2              Total Time: max(T1, T2)
```

### 5. Получение отдельных последовательностей (строки 25-26)
```javascript
const sequence1 = context.getSequence();
const sequence2 = context.getSequence();
```
- Извлекает два отдельных объекта последовательности из контекста
- Каждая последовательность поддерживает своё состояние
- Они могут использоваться независимо для разных разговоров

### 6. Создание отдельных сессий (строки 28-33)
```javascript
const session1 = new LlamaChatSession({
    contextSequence: sequence1
});
const session2 = new LlamaChatSession({
    contextSequence: sequence2
});
```
- Создаёт чат-сессию для каждой последовательности
- У каждой сессии своя история разговоров
- Сессии полностью независимы
- В этом примере нет системных промптов (можно добавить)

### 7. Определение вопросов (строки 35-36)
```javascript
const q1 = "Hi there, how are you?";
const q2 = "How much is 6+6?";
```
- Два完全不同ных вопроса
- Будут обработаны одновременно
- Разные типы: разговорный vs вычислительный

### 8. Параллельное выполнение с Promise.all (строки 38-44)
```javascript
const [
    a1,
    a2
] = await Promise.all([
    session1.prompt(q1),
    session2.prompt(q2)
]);
```

**Как это работает:**

1. `session1.prompt(q1)` начинается асинхронно
2. `session2.prompt(q2)` начинается асинхронно (не ждёт первого)
3. `Promise.all()` ждёт завершения ОБЕИХ
4. Возвращает результаты в массиве: [response1, response2]
5. Деструктуризует в `a1` и `a2`

**Ключевое преимущество**: Оба промпта обрабатываются одновременно, а не один за другим!

### 9. Отображение результатов (строки 46-50)
```javascript
console.log("User: " + q1);
console.log("AI: " + a1);

console.log("User: " + q2);
console.log("AI: " + a2);
```
- Выводит обе пары вопрос-ответ
- Результаты появляются по порядку, несмотря на параллельную обработку

## Продемонстрированные ключевые концепции

### 1. Параллельная обработка

Вместо:
```javascript
// Sequential (slow)
const a1 = await session1.prompt(q1);  // Wait
const a2 = await session2.prompt(q2);  // Wait again
```

Мы используем:
```javascript
// Parallel (fast)
const [a1, a2] = await Promise.all([
    session1.prompt(q1),
    session2.prompt(q2)
]);
```

### 2. Контекстные последовательности
Контекст может хранить несколько независимых последовательностей:

```
┌─────────────────────────────────────┐
│          Context (Shared)           │
│  ┌───────────────────────────────┐  │
│  │  Model Weights (8B params)    │  │
│  └───────────────────────────────┘  │
│                                     │
│  ┌─────────────┐  ┌─────────────┐  │
│  │ Sequence 1  │  │ Sequence 2  │  │
│  │ "Hi there"  │  │ "6+6?"      │  │
│  │ History...  │  │ History...  │  │
│  └─────────────┘  └─────────────┘  │
└─────────────────────────────────────┘
```

## Сравнение производительности

### Последовательное выполнение
```
Request 1: 2 seconds
Request 2: 2 seconds
Total: 4 seconds
```

### Параллельное выполнение (этот пример)
```
Request 1: 2 seconds ──┐
Request 2: 2 seconds ──┤ Both running
Total: ~2 seconds      └─ simultaneously
```

**Ускорение**: ~2x для 2 последовательностей, масштабируется с увеличением количества последовательностей

## Случаи использования

### 1. Мульти-пользовательские приложения
```javascript

    session1.prompt(user1Query),
    session2.prompt(user2Query),
    session3.prompt(user3Query)

```

### 2. Мультиагентные системы

```javascript
const [

    plannerSession.prompt("Plan the task"),

```

### 3. Бенчмаркинг
```javascript
// Test multiple prompts for evaluation
const results = await Promise.all(

```

### 4. A/B тестирование

## Соображения по ресурсам

### Использование памяти
Каждая последовательность нуждается в памяти для:
- Истории разговоров
- Промежуточных вычислений
- KV-кэша (кэш ключ-значение для внимания трансформера)

**Правило**: Больше последовательностей = больше памяти

### Утилизация GPU
- **Одна последовательность**: Может недогружать GPU
- **Несколько последовательностей**: Лучшая утилизация GPU
- **Слишком много последовательностей**: Может превысить VRAM, вызывая замедление

### Оптимальное количество последовательностей
Зависит от:
- Доступного VRAM
- Размера модели
- Длины контекста
- Размера батча

**Обычно**: 2-8 последовательностей для потребительских GPU

## Ограничения и соображения

### 1. Общий лимит контекста
Все последовательности разделяют один пул памяти контекста:
```
Total context size: 8192 tokens
Sequence 1: 4096 tokens
Sequence 2: 4096 tokens
Maximum distribution!
```

### 2. Не настоящий параллелизм для CPU
На системах только CPU последовательности чередуются, а не работают параллельно. Всё равно обеспечивает лучшую общую пропускную способность.

### 3. Накладные расходы загрузки модели
Модель загружается один раз и разделяется, что эффективно. Но начальная загрузка всё равно занимает время.

## Почему это важно для AI-агентов

### Эффективность в продакшене
Реальные системы агентов должны:
- Обрабатывать несколько запросов параллельно
- Быстро отвечать пользователям
- Эффективно использовать оборудование

### Мультиагентные архитектуры
Сложные системы агентов часто имеют:
- **Агент планирования**: Думает о стратегии
- **Агент исполнителя**: Выполняет действия
- **Агент критика**: Оценивает результаты

Они могут работать параллельно, используя отдельные последовательности.

### Масштабируемость
Этот паттерн является фундаментом для:
- Веб-сервисов с несколькими пользователями
- Пакетной обработки данных
- Распределённых систем агентов

## Лучшие практики

1. **Соотносите последовательности с нагрузкой**: Не создавайте больше, чем нужно
2. **Мониторьте использование памяти**: Каждая последовательность потребляет VRAM
3. **Используйте подходящий размер батча**: Баланс между скоростью и памятью
4. **Очищайте ресурсы**: Всегда освобождайте после использования
5. **Обрабатывайте ошибки**: Оборачивайте Promise.all в try-catch
