# Объяснение кода: `tree-of-thought.js`

Это руководство следует фактической структуре кода, чтобы Вы могли сопоставить каждую концепцию ToT с конкретными функциями.

## Запуск

```bash
node examples/12_tree-of-thought/tree-of-thought.js
```

---

## 1) Настройка: модель, схемы, константы

В верхнем уровне файла:

- `HYPOTHESIS_TYPES` определяет четыре конкурирующие ветки.
- `BEHAVIOR_INPUT` — описание случая.
- `hypothesisSchema`, `scoreSchema`, `rankingSchema`, `analysisSchema` определяют JSON-контракты для каждой фазы.
- `promptJson(schema, userText)` — общая утилита, которая:
  - сбрасывает историю чата,
  - обеспечивает грамматику схемы,
  - парсит/исправляет JSON.

Это позволяет каждой функции фазы фокусироваться на логике, а не на шаблонном коде парсера.

---

## 2) Фаза 1 (Ветвление): `developHypothesis()`

`developHypothesis(behavior, hypothesisType)` делает одну вещь:

- просит модель порассуждать ровно через одну призму,
- возвращает структурированный объект:
  - `name`
  - `argument`
  - `signals`
  - `counter_evidence`

В `runTreeOfThoughtMotivationAnalysis()` это запускается в цикле по `HYPOTHESIS_TYPES`, создавая четыре конкурирующие ветки.

---

## 3) Фаза 2 (Оценка): `scoreHypothesis()` + `rerankHypotheses()`

### Сырая оценка по веткам

`scoreHypothesis(behavior, hypothesis)` возвращает:

- `score` (сырое числовое значение из формулы),
- `details` (`explanatory_power`, `plausibility`, `falsifiability`),
- `blindSpot`,
- `reasoning`.

### Калибровка против ничьей

`rerankHypotheses(behavior, scoredHypotheses)` вынуждает строгое ранжирование без ничьих, затем отображает ранги на калиброванные оценки:

- rank1 -> `8.8`
- rank2 -> `8.1`
- rank3 -> `7.4`
- rank4 -> `6.7`

Вот почему консоль показывает:

- захваченные сырые оценки
- затем калиброванные оценки, используемые для обрезки

Так учащиеся видят, что система *фактически* использует для выбора веток.

---

## 4) Фаза 3 (Обрезка): `pruneHypotheses()`

`pruneHypotheses(scoredHypotheses)`:

- сортирует по убыванию оценки,
- сохраняет победителя,
- возвращает **отброшенные** ветки.

Это структурное ядро ToT в этом примере: один победитель продолжает, альтернативы отбрасываются.

---

## 5) Фаза 4 (Заключение): `createConclusion()`

`createConclusion(behavior, winner)` строит финальный анализ, используя только:

- имя победителя
- аргумент победителя
- сигналы победителя

Отброшенные ветки не влияют на финальный ответ.  
Это намеренное ограничение показано в консольном блоке: `WHAT TOT LOST IN THIS RUN`.

---

## 6) Поток оркестрации: `runTreeOfThoughtMotivationAnalysis()`

Эта функция — сквозной контроллер:

1. ветвление (сбор гипотез)
2. оценка (сырая + калиброванная)
3. обрезка (победитель + отброшенные)
4. заключение (только победитель)
5. вывод результата + вызов утилиты визуализации

Визуализация намеренно делегирована:

- `writeToTMotivationVisualization(...)`

чтобы файл примера оставался сфокусированным на потоке управления ToT.

---

## Рекомендуемый порядок чтения кода

Читайте функции в этой последовательности:

1. `promptJson`
2. `developHypothesis`
3. `scoreHypothesis`
4. `rerankHypotheses`
5. `pruneHypotheses`
6. `createConclusion`
7. `runTreeOfThoughtMotivationAnalysis`

Этот порядок отражает поток выполнения и делает файл гораздо более понятным.
