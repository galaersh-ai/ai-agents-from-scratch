## Маршрутизация инструментов с эмбеддингами и примерами

**Идея:** Перед вызовом chat-модели с `functions` Вы **предварительно фильтруете**, какие инструменты она увидит. **Маленькая модель эмбеддингов** оценивает, насколько близко пользовательское сообщение к коротким **фразам-примерам** для каждого инструента (типичные намерения). Вы сохраняете только top-k инструментов (и опционально инструменты **«всегда включаемые»**), затем запускаете обычный цикл вызова инструментов.

---

### Мотивация: почему не давать LLM все инструменты каждый раз?

Представьте агента IT-помощника с 12 инструментами сегодня, которые вырастут до 80 инструментов в следующем квартале:

- **Стоимость контекста:** Каждый инструмент отправляет имя, описание и JSON-схему в промпт. 80 инструментов могут добавить тысячи токенов шума на каждый запрос.
- **Неправильные вызовы:** Большие каталоги увеличивают вероятность того, что модель выберет правдоподобную, но неправильную функцию.
- **Латентность:** Менее объёмные полезные нагрузки инструментов могут означать более быстрое поведение первого токена на некоторых стеках.

Маршрутизация — дешёвый предварительный фильтр. Chat-модель всё ещё решает **аргументы** и **когда** вызывать.

---

### Поток

```text
User message
    → embed (small model, e.g. bge-small-en)
    → cosine similarity vs precomputed exemplar embeddings (one per tool phrase)
    → scoreTools()  →  Map<toolKey, maxSimilarity>
    → selectToolKeys(scores, k, alwaysInclude)  →  Set<toolKey>
    → session.prompt(..., { functions: subset })
```

Два шага маршрутизации намеренно разделены на отдельные функции, чтобы каждый был легко инспектируем и тестируем независимо.

Используйте **одну и ту же** контрольную точку эмбеддингов для запросов и примеров. Смешивание моделей ломает сходство.

---

### Оценка примеров

Каждый инструмент имеет несколько строк-примеров. Оценка для инструмента — это **максимальное** косинусное сходство между запросом и любым из примеров этого инструмента. Максимум лучше среднего: одно сильное совпадение примера сохраняет инструмент конкурентоспособным, даже если другие промахиваются.

**Хорошие примеры — не копии текста пользователя.** В продакшене Вы собираете парафразы, тикеты поддержки и варианты FAQ. Если Ваши примеры идентичны живым запросам, маршрутизация выглядит «магической», но Вы не тестируете обобщение — и переобучитесь на одной формулировке.

---

### Режимы отказа

1. **Сбой recall:** Правильный инструмент набирает оценку ниже порога. Ассистент может отказаться или галлюцинировать.
2. **Разрыв примеров:** Формулировка пользователя не похожа ни на одну сохранённую фразу (отраслевой жаргон, опечатки, другой язык).
3. **Мульти-намерение, k слишком мало:** Одно сообщение требует два инструмента, но k=1 показывает только доминирующий кластер. *Демо-случай 5 показывает это намеренно.*
4. **Ложные срабатывания:** Два несвязанных инструмента разделяют расплывчатые формулировки («проверить», «статус», «соединение»).

**Смягчения:**
- Больше/лучшие примеры, покрывающие пограничные формулировки
- Больший k (с убывающей отдачей по мере возврата шума каталога)
- **Множества «всегда включаемых»** для критичных инструментов (комплаенс, логирование, безопасность), которые всегда должны быть в области
- Гибридные ключевые слова как быстрый предварительный проход
- Ранжировщик второго этапа для точности по короткому списку

---

### «Всегда включаемые»: закрепление инструментов независимо от оценки

Некоторые инструменты всегда должны быть в области:

- **Инструмент комплаенса**, который каждое взаимодействие должно предлагать (например, `createSupportTicket`)
- **Инструмент безопасности**, который должен запускаться при появлении определённых тем
- **Инструмент контекста**, который агенту нужен на каждом ходу (например, `getUserProfile`)

Закреплённые инструменты объединяются с множеством **после** извлечения, поэтому они не потребляют слот извлечения. Демо-случай 6 показывает это: k=1 извлекает только VPN-инструмент, но `getUserAccount` закреплён, поэтому оба доступны.

### Какая модель эмбеддингов?

Для локального пути этого репозитория **только node-llama-cpp**: **[bge-small-en-v1.5](https://huggingface.co/CompendiumLabs/bge-small-en-v1.5-gguf)** в формате GGUF (Q8_0). Он маленький, англоориентированный и хорошо документирован вместе с `createEmbeddingContext`.

Альтернативы: более лёгкая квантизация того же семейства для меньшего потребления RAM; **многоязычная** модель эмбеддингов GGUF, если намерения пользователей не на английском — всё равно всегда используйте одну модель для примеров и запросов.

### Детали chat-стека (Qwen3 + инструменты)

С дефолтным шаблоном чата Qwen модель может написать видимый пользователю ответ после вызова инструмента только внутри внутренних сегментов «мышления». `session.prompt()` возвращает сегменты plain-text, поэтому Вы можете получить **пустую строку**, даже если инструмент отработал. Использование `QwenChatWrapper` с `thoughts: "discourage"` избегает этой ловушки.

### Ключевой вывод

Маршрутизация инструментов — это **RAG для инструментов**: примеры — документы, пользовательское сообщение — запрос, а косинусное сходство — извлечение. Она не заменяет тщательный дизайн инструментов или суждение chat-модели — но она масштабирует каталоги агентов так же, как извлечение масштабирует базы знаний.

```text
Without routing:  12 tools → 12 schemas in prompt (fine)
                  80 tools → 80 schemas in prompt (noisy, expensive)
                 500 tools → impractical

With routing:     500 tools, k=4 → only 4 schemas in prompt every time
```
