# Проектирование промптов

Проектирование промптов — самый быстрый и простой способ определить поведение агента: его личность, функцию и выбор (например, когда использовать инструменты). Агенты работают с двумя категориями промптов: системными и пользовательскими.

Пользовательские промпты — это сообщения, которые люди вводят во время разговора. Они меняются с каждым взаимодействием и не контролируются разработчиком.

Системные промпты содержат инструкции, установленные разработчиками, которые остаются неизменными на протяжении диалога. Они определяют тон агента, возможности, ограничения и руководства по использованию инструментов.

Ознакомьтесь с системными промптами Anthropic:
https://docs.claude.com/en/release-notes/system-prompts#september-29-2025

## Проектирование промптов

При создании промптов для агентов Вам нужно достичь двух вещей:

1. Заставить агента хорошо решать задачи
   - Помочь ему правильно выполнять сложные задачи
   - Обеспечить ясное, логическое мышление
   - Снизить количество ошибок

2. Сохранять согласованность личности агента
   - Определить, кто агент и как он говорит
   - Соответствовать голосу Вашего бренда
   - Отвечать с соответствующей эмоцией для каждой ситуации

Обе цели одинаково важны. Точный ответ, delivered rudely, ухудшает пользовательский опыт. Дружелюбный ответ, который не помогает, бесполезен.

## Стратегии промптов

### Роль агента
Присвоение LLM конкретной роли улучшает её ответы — она естественно adoptирует словарный запас и экспертизу этой роли.

Примеры:
- «Вы педиатр» → использует медицинские термины, обсуждает развитие детей, рекомендует age-appropriate лечения
- «Вы шеф-повар» → объясняет техники готовки, предлагает замены ингредиентов, обсуждает профили вкусов
- «Вы учитель математики в средней школе» → разбивает задачи пошагово, использует простой язык, даёт примеры для практики
- «Вы основатель стартапа» → фокусируется на росте, использует бизнес-метрики, думает о масштабируемости

Делайте роли конкретными:
Вместо: «Вы писатель»
Лучше: «Вы технический блогер, который упрощает сложные концепции AI для новичков»

Роли лучше всего работают для специализированных вопросов и должны быть установлены в системных промптах.

### Будьте конкретны, а не расплывчаты
LLM интерпретируют инструкции буквально. Расплывчатые промпты дают случайные результаты. Конкретные промпты дают согласованные ответы.

Примеры расплывчатых vs конкретных:
❌ Расплывчато: «Напишите что-нибудь о собаках»
✅ Конкретно: «Напишите руководство из 3 абзацев о дрессировке щенка команде «сидеть»»

❌ Расплывчато: «Сделайте лучше»
✅ Конкретно: «Исправьте грамматические ошибки и сократите до 100 слов»

❌ Расплывчато: «Будьте профессиональны»
✅ Конкретно: «Используйте формальный язык, избегайте сокращений, обращайтесь к читателю на «Вы»»

❌ Расплывчато: «Проанализируйте эти данные»
✅ Конкретно: «Найдите 3 главных тренда и объясните, что вызвало каждый из них»

Почему это важно: У LLM тысячи способов интерпретировать расплывчатые инструкции. Она угадает, чего Вы хотите — и часто угадает неправильно. Ясные инструкции устраняют необходимость угадывать и дают Вам контроль над выводом.

Правило: если человеческому помощнику потребовалось бы задать уточняющие вопросы, Ваш промпт слишком расплывчат.

### Структурирование ввода LLM с помощью JSON
Использование JSON для структурирования ввода помогает LLM понимать задачи яснее и упрощает интеграцию. Вместо отправки сплошного текста разбейте запрос на подписанные части: task, input, constraints, output_format.

Преимущества:
- Ясность: ключи JSON показывают модели, что означает каждая часть.
- Надёжность: проще парсить и валидировать ответы.
- Согласованность: снижает случайные или нарративные ответы.
- Интеграция: хорошо работает с API и схемами.

Лучшие практики:
- Держите просто и плоско — избегайте глубокой вложенности.
- Используйте описательные ключи («task», «context», «constraints»).
- Укажите модели точный формат вывода (например, «Отвечайте только валидным JSON»).
- Опционально определите JSON Schema для обеспечения структуры.
- Всегда валидируйте ответ в Вашем коде.

Пример:
````
{
  "task": "summarize",
  "input_text": " - Текст статьи здесь. - ",
  "constraints": {
    "max_words": 100,
    "audience": "non-technical"
  },
  "output_format": {
    "type": "JSON",
    "schema": {
      "summary": "string",
      "key_points": ["string"]
    }
  }
}
````

Этот структурированный формат помогает модели отделить что делать, какие данные использовать и как отвечать, что приводит к более согласованным, машиночитаемым выводам.

### Few-Shot промптинг

Few-shot промптинг означает предоставление LLM нескольких примеров того, что Вы хотите, перед тем как просить выполнить новую задачу.

Это как показать ученику две или три решённые задачи, чтобы он понял паттерн.

Пример:
```
Пример 1:
Отзыв: «Номер был чистый и тихий.»
Категория: Положительный

Пример 2:
Отзыв: «Персонал был груб и неконтактен.»
Категория: Отрицательный

Пример 3:
Отзыв: «Завтрак был нормальный, но кофе был холодный.»
Категория: Нейтральный

Теперь классифицируйте:
Отзыв: «Вид с балкона был потрясающий!»
Категория:
```

Модель учится на примерах и продолжает в том же стиле — здесь она ответит: «Положительный».

Few-shot промпты полезны, когда Вы хотите согласованный тон, формат или логику без дообучения модели.

### Цепочка рассуждений

Цепочка рассуждений означает просьбу к LLM думать пошагово, а не сразу переходить к ответу.

Это помогает модели лучше рассуждать, особенно для логических, математических или многошаговых задач.

Пример:
Вопрос: Если 3 яблока стоят $6, сколько стоят 5 яблок?
Давайте думать пошагово.

Рассуждение модели:
3 яблока → $6 → каждое яблоко стоит $2.
5 яблок × $2 = $10.

Ответ: $10

Поощряя пошаговое мышление, вы помогаете модели делать меньше ошибок и ясно объяснять свои рассуждения.
